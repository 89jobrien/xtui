# xtui — cargo-xtask TUI Dashboard

**Date:** 2026-06-07
**Status:** Design approved, pending implementation

## Goal

A lightweight ratatui dashboard that discovers cargo-xtask commands in any
Rust workspace, presents them as a selectable list, runs them, and renders
captured output — all in a split-pane TUI.

## Architecture

Standalone binary crate at `~/dev/xtui`. No workspace dependencies on
other projects. Points at any Rust workspace via CLI arg or CWD detection.

### Crate Structure

```
xtui/
  src/
    main.rs          -- CLI entry, terminal setup/teardown
    app.rs           -- App state, event loop, key handling
    discover.rs      -- Command discovery (help parse + source fallback)
    runner.rs        -- Async command execution + output capture
    ui.rs            -- ratatui widget layout and rendering
  Cargo.toml
  docs/plans/
    2026-06-07-xtui.md
```

Single crate, no sub-crates. YAGNI.

### Dependencies

| Crate       | Version | Purpose                                      |
| ----------- | ------- | -------------------------------------------- |
| ratatui     | latest  | TUI framework (re-exports crossterm backend) |
| tokio       | 1       | Async command execution                      |
| tokio-util  | latest  | LinesCodec for line-by-line output streaming |
| ansi-to-tui | 8       | ANSI escape code rendering in output pane    |
| anyhow      | 1       | Error handling                               |
| regex       | 1       | Help output + source parsing                 |

Note: No direct `crossterm` dependency needed -- ratatui 0.30+ re-exports
the crossterm backend via `ratatui-crossterm`. Resolve `latest` versions
via `cargo add`.

### Data Flow

```
CLI arg or CWD
  |
  v
discover::discover_commands(workspace_root)
  |-- try: `cargo xtask` (no args) -> parse stderr/stdout lines
  |-- fallback: read xtask/src/main.rs -> extract match arms
  |
  v
Vec<XtaskCommand { name: String, description: Option<String> }>
  |
  v
app::App { commands, selected, output, running }
  |
  v
Event loop:
  - Render: ui::draw(frame, &app)
      Left pane:  selectable list of commands
      Right pane: scrollable output (or "select a command")
  - Input:
      j/k or Up/Down  = navigate list
      Enter            = run selected command
      q / Ctrl-C       = quit
      Esc              = cancel running command / clear output
```

### Layout

Split-pane (option A from brainstorm):

```
+------------------+------------------------------------+
| Commands    [3]  | Output                             |
|                  |                                    |
|   verify         | $ cargo xtask verify               |
| > lint           | Compiling minibox v0.30.0          |
|   fix            | Checking minibox-core v0.30.0      |
|   pre-commit     | ...                                |
|   test-unit      |                                    |
|   coverage       |                                    |
|                  |                                    |
+------------------+------------------------------------+
```

Left pane: 30% width (min 20 cols). Right pane: remainder.
Status bar at bottom: workspace name, running/idle, command count.

### Command Discovery (`discover.rs`)

**Primary: help output parsing**

```
cargo xtask 2>&1
```

Parse each non-empty line matching `^\s+(\S+)\s+(.*)$` as
`(command_name, description)`. Skip lines that don't match (headers,
blank lines). This works for minibox (eprintln-based help), braid
(bail-based error), and most conventions.

**Fallback: source parsing**

Read `xtask/src/main.rs`. Extract `Some("command-name")` patterns from
match arms via regex: `Some\("([^"]+)"\)`. Description set to `None`.

**Edge cases:**

- Nested subcommands (e.g. `borrow fixtures`) — flatten as
  `borrow-fixtures` or skip; MVP treats each match arm as one command.
- xtask not present — show error in output pane, empty command list.
- xtask fails to compile — show compiler error in output pane.

### Command Execution (`runner.rs`)

- Spawn `cargo xtask <command>` via `tokio::process::Command`
- Pipe stdout and stderr separately (`Stdio::piped()`)
- Spawn two async tasks, each wrapping its pipe in
  `FramedRead<_, LinesCodec>` (from `tokio-util`), sending lines into a
  shared `tokio::sync::mpsc` channel -- this interleaves output
  approximately in arrival order
- App polls channel each tick and appends to output buffer
- Running state tracked: `App.running: Option<Child>` (allows Esc cancel
  via `child.kill()`)
- Output buffer capped at 10,000 lines (ring buffer or truncate head)

### UI Rendering (`ui.rs`)

- Left pane: `List` widget with highlight style
- Right pane: `Paragraph` widget with `Wrap { trim: false }`, auto-scroll
  to bottom while running, manual scroll (Shift+j/k or PgUp/PgDn) when
  idle
- Status bar: `Paragraph` with workspace path, state indicator
- Colors: default terminal colors, highlight via reverse video. No
  theme system.

## Tech Decisions

| Decision                       | Rationale                                        |
| ------------------------------ | ------------------------------------------------ |
| Standalone crate, not plugin   | Zero coupling, works with any workspace          |
| Help-first discovery           | Works regardless of xtask code structure         |
| Source fallback                | Handles xtasks that exit non-zero with no args   |
| Interleaved stdout+stderr      | Two async readers into mpsc; arrival-order merge |
| tokio for async                | Need non-blocking child process + line streaming |
| Split pane (not overlay/modal) | Always-visible context; simplest mental model    |
| ratatui 0.30                   | Latest stable, matches kan                       |

## Out of Scope

- Persistent config or settings file
- Command history or favorites
- Multi-workspace aggregation (run against one workspace at a time)
- Custom keybinding configuration
- Command argument editing (runs commands as-is, no arg prompts)
- Filtering or search within the command list (MVP; add later if needed)
- Color/theme customization

## Resolved Questions

1. **Workspace detection** — auto-detect from CWD (walk up looking for
   `Cargo.lock` + `xtask/` dir) with optional `--path <dir>` override.

2. **Nested subcommands** — skip for MVP. Only single-word commands
   from help output or source match arms.

3. **ANSI color passthrough** — render from day one using `ansi-to-tui`
   crate (v8). Output pane uses `ansi_to_tui::ansi_to_text()` to
   convert raw ANSI output into ratatui `Text`.

4. **Refresh/re-discover** — `r` keybinding to re-run discovery.

5. **Exit code display** — show in status bar after command finishes.
